## Purpose

The Workspace Surface is the focused execution environment for a specific object or workflow. It is where the user goes when they need **depth**, not just awareness.

A workspace is the "Workbench" of the system, providing all the context and controls required to complete a complex task.

---

## 1. Surface Characteristics

-   **High Context Density**: Unlike the Daily Feed's brevity, the Workspace shows all the available data for the object (e.g., student history, syllabus map, fee ledger).
-   **Clear Action Ladder**: Primary actions (e.g., "Mark Attendance") are prioritized, while secondary actions (e.g., "View Attendance History") are reachable but not cluttering.
-   **Recommendations + Manual Control**: The system's suggestions (Agent Drafts, Recovery Paths) are presented alongside the manual controls, allowing the user to compare the two.
-   **State & History**: A visible trail of what has already happened with this object (e.g., previous lessons, parent responses).

---

## 2. Core Workspace Examples

| Workspace | Primary Persona | Goal |
| :--- | :--- | :--- |
| **Class Teaching** | Teacher | Executing a specific period, marking attendance, and logging syllabus progress. |
| **Student Intervention** | Teacher/Principal | Reviewing a student's risk profile and drafting a parent-facing support plan. |
| **Event Console** | Admin/Principal | Managing a field trip or competition across notices, consents, and payments. |
| **Setup & Calibration** | Admin | Completing the five-layer institutional substrate and closing confidence gaps. |
| **Notice Draft** | Admin/Principal | Segmenting audiences and drafting high-sensitivity institutional communication. |

---

## 3. Structural Components

Every Workspace surface should contain:

1.  **The Context Bar**: Fixed at the top, showing the object name (e.g., "Grade 5 History") and its current state.
2.  **The Primary Stage**: The central area for the main action (e.g., Attendance Roster, Notice Draft).
3.  **The Intelligence Sidebar**: Contextual signals, recommendations, and "Agent Nudges" related to this specific object.
4.  **The History/Traceability Panel**: A quick-access log of recent actions and outcomes for the object.

---

## 4. Interaction Pattern: "The Comparison View"

When the system provides a recommendation (e.g., a substitution plan), the Workspace should allow the user to see the **System's Plan** side-by-side with the **Manual Plan**, highlighting the data-driven rationale for the system's choice.

---

## 5. Design Rule: "Contextual Depth, Not Clutter"

A Workspace should only show the data required for the task at hand. For example, a Teacher in the **Class Workspace** should see student attendance history but not their full fee ledger unless it's relevant to a specific "Ad-hoc Payment" event.
