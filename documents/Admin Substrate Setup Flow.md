## Architectural Purpose

This document defines the **Admin Substrate Setup Flow**. 

Before a Principal can click "Approve," and before a Teacher can enter a grade, the system must learn the physical and legal constraints of the entire institution. This is the domain of the **System Administrator**. 

This flow details how the Admin teaches the AI the rules of the school (The Operational Substrate).

---

## 1. The Entrance: The Mapping Canvas

The Admin logs in. They do not see the standard `Role-based Dashboard` filled with active students. They see the **System Console**, an ultra-dense, developer-like interface optimized for structural engineering.

### UX State 1: The Blueprint
*   **The Aesthetic**: Unlike the Teacher's breathable layout, the Admin interface deliberately uses high-density `#2d3335` dark modes and `Inter` tabular typography to differentiate system-level architecture from daily human operations.
*   **The Zones**: The screen is divided into massive conceptual blocks: `Timetable Matrix`, `User Ontology`, `Spatial Geography (Rooms)`, and `Policy Weights`.

---

## 2. Ingestion: Teaching the AI

The Admin does not program the AI using code. They define boundaries using "No-Code" interlocking Rulesets. 

### UX State 2: Defining Geography and Time
1.  **The Inputs**: The Admin imports the basic variables: *We have 40 rooms. We have 60 teachers. The day starts at 8:00 AM. Period 1 is 45 minutes.*
2.  **The Physical Constraints**: They establish hard physical laws. *Teacher Y cannot climb stairs.* *Room 12B only holds 20 desks.* 

### UX State 3: Defining the Trust & Policy Layer (The sliders)
This is where Mintrix truly differentiates. The Admin sets the `Autonomy Ceilings`.
*   **The Interface**: A massive screen filled with horizontal sliders, ranging from `0` (Manual Human Only) to `100` (Full AI Autonomy).
*   **The Policy**: "Attendance Triage." The Admin slides it to `80`. This legally permits the `Operator Agent` to auto-request Substitute Teachers when someone calls in sick, but explicitly forbids it from officially canceling a class without the Principal's Amber Signature.

---

## 3. The Validation: The System Simulation

Once the Admin inputs the rules, they do not blindly hit `Save to Production`. Before the system goes live, Mintrix forces a Stress Test.

### UX State 4: The Simulation Engine
*   **The Action**: The Admin clicks `Run Simulation`. 
*   **The Computation**: The AI artificially simulates 1 month of a school's life playing out thousands of scenarios (Fire drills, mass sickness, timetable clashes) strictly against the newly drawn rules.
*   **The Output**: The system returns a massive `Readiness Report`. It explicitly flags `Double-Bindings` (e.g., *"Warning: You scheduled 3rd Grade recess at the same time as 12th Grade AP Physics in adjacent zones. Noise bleed is calculated at 85% disruption. Resolve before launching."*)

---

## 4. The Lock-In

Once the Simulation glows green, the Admin hits `Deploy Substrate`.

*   **The Final UI State**: The developer-dark screen fades out. The system flashes a `Signature Gradient` Blue confirmation. 
*   **The Awakening**: The AI's neural weights are now permanently tuned to this specific school's physics and laws. The system unlocks the Principal, Teacher, and Parent accounts, dropping them into a perfectly mapped `"Live" State`.
